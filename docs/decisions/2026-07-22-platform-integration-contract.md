# สัญญาการเชื่อมต่อกับแพลตฟอร์ม (AI for Thai และที่อื่น)

- วันที่ 2026-07-22 +07:00
- สถานะ Implemented (2026-07-22)
- บริบท ตอนเขียน worker (21 ก.ค.) เราตั้งสมมติฐานว่าแพลตฟอร์มจะประกาศ spec ของคิวออกมา
  แล้วเราค่อยปรับตาม สมมติฐานนั้นถูกยกเลิก **เราเป็นคนนิยามสัญญาเอง** และรูปแบบการติดตั้ง
  มีได้หลายแบบ เอกสารนี้คือสัญญานั้น ไม่ใช่การเดาว่าเขาต้องการอะไร

## ทำไมต้องมีเอกสารนี้

`app/worker/handler.py` เขียนกำกับตัวเองไว้ว่า schema เป็น `provisional` เพราะรอ spec
การรอนั้นทำให้เกิดสองปัญหา หนึ่ง ไม่มีใครรู้ว่าอะไรคือของที่เรารับประกัน สอง เราออกแบบ
ตามสิ่งที่เดาว่าอีกฝ่ายต้องการ แทนที่จะออกแบบตามคำสัญญาของโปรดักต์เราเอง

เอกสารนี้กลับด้าน เรานิยามสิ่งที่เราให้และรับประกัน ใครจะต่อเข้ามาก็ต่อตามนี้ ถ้าฝั่งเขา
ต่างออกไป สิ่งที่ต้องแก้คือชั้น transport ชั้นเดียว ไม่ใช่แกนของระบบ

## 1 หน้าสัญญา (declared surface)

รับประกันความเสถียรเฉพาะ 5 endpoint นี้

| Endpoint | หน้าที่ |
|---|---|
| `GET /api/health` | สถานะ เวอร์ชัน และความสามารถที่เปิดอยู่ |
| `POST /api/sanitize` | ปกปิด PII คืนข้อความที่ปกปิดแล้วกับ `session_id` |
| `POST /api/reidentify` | คืนค่าจริงกลับเข้าไปในข้อความ ด้วย `session_id` |
| `POST /api/analyze` | รายงานความเสี่ยง PDPA คะแนน เกรด และหมวดมาตรา 26 |
| `POST /api/guard` | สัญญาณเตือน prompt injection แบบ dependency-light (explicit rules + bounded intent classifier) |

endpoint อื่นทั้งหมด (`/api/detect`, `/api/roundtrip`, `/api/redact-pdf`,
`/api/analyze-report`, `/demo`, control plane) ถือเป็น **ภายใน** มีไว้ให้ extension
เดสก์ท็อป หรือ demo ใช้ เปลี่ยนได้โดยไม่ถือว่าผิดสัญญา

กติกาการเปลี่ยนแปลง **เพิ่ม endpoint หรือเพิ่มฟิลด์ใน response ได้ ถอดหรือเปลี่ยน
ความหมายของที่ประกาศไว้ไม่ได้** `/api/health` คืนฟิลด์ `contract_version` (จำนวนเต็ม
เริ่มที่ `1`) แยกจาก `version` ที่เป็นเวอร์ชันของซอฟต์แวร์ เพื่อให้ผู้เรียกตรวจได้เองว่า
คุยกับสัญญารุ่นไหนอยู่ ตัวเลขนี้ขยับเมื่อมีการเปลี่ยนแบบทำลายความเข้ากันได้เท่านั้น
ซึ่งตามกติกาข้างต้นแปลว่าแทบไม่ควรขยับเลย

## 2 mapping (คู่โทเคนกับค่าจริง) อยู่ที่ไหน

**ค่าเริ่มต้นคือ session mode** `/api/sanitize` คืน `session_id` ส่วน mapping อยู่ใน
หน่วยความจำของ process เท่านั้น (`SessionService` มี LRU cap, idle TTL, ล้างด้วย null byte
ตอน drop หรือ evict) การคืนค่าทำผ่าน `/api/reidentify` ด้วย `session_id` เดิม

**ไม่มี endpoint ใดในหน้าสัญญาที่คืน mapping ออกไป** ข้อนี้เป็นคุณสมบัติเชิงโครงสร้าง
ไม่ใช่คำขอร้อง ผู้เรียกจึงไม่มีทางเผลอ log ค่าจริงจากผลลัพธ์ของเรา

ผลที่ตามมาที่ต้องยอมรับ ถ้ารันหลาย instance ต้องส่งคำขอของ session เดียวกันไปที่ instance
เดิม (sticky routing) และถ้า container รีสตาร์ต mapping จะหาย ซึ่งถูกต้องตามการออกแบบ
เพราะเราสัญญาว่าไม่เขียน PII ลงดิสก์

## 3 การยืนยันตัวตนของผู้เรียก

env `AIGUARD_API_KEY`

- **ไม่ตั้ง** พฤติกรรมเดิมทุกไบต์ ไม่มีการบังคับ header (เส้นทาง extension ในเครื่องผู้ใช้
  ที่ไม่มีช่องส่ง token ยังทำงานเหมือนเดิม)
- **ตั้ง** ทั้ง 4 endpoint ที่ไม่ใช่ health บังคับ header `X-AIGuard-Key` เทียบด้วย
  `secrets.compare_digest` ไม่ตรงหรือไม่มีคืน 401

`/api/health` ไม่บังคับ key เพราะ orchestrator ต้องใช้ตรวจว่ายังมีชีวิต และมันเปิดเผยแค่
สถานะ เวอร์ชัน กับรายการความสามารถ ไม่มีข้อมูลผู้ใช้

ตอน boot ถ้าไม่ได้ตั้ง key ให้เขียน log เตือนระดับ warning ให้ชัด การ deploy ทุกแบบใน
คู่มือต้องตั้งค่านี้เสมอ

ใช้แพตเทิร์นเดียวกับ `AIGUARD_TOKEN` ที่ควบคุม control plane อยู่แล้ว คือความปลอดภัยมาจาก
การตั้งค่า ไม่ใช่การ build image คนละก้อน

## 4 รูปแบบการติดตั้งที่รองรับ

ทั้งสองแบบใช้ image เดียวกัน

**แบบที่ 1 HTTP service** (แนะนำเป็นค่าเริ่มต้น) container ฟังพอร์ต 8000 แพลตฟอร์มชี้
gateway เข้ามาเรียก 5 endpoint ตามสัญญา

**แบบที่ 2 queue worker** `python -m app.worker` ดึงงานจากคิวแล้วส่งผลกลับ ส่วนที่เดาไม่ได้
ถูกกักไว้ในไฟล์ `transport.py` ไฟล์เดียว เปลี่ยนรูปคิวคือแก้ไฟล์นั้น

base image ถูก pin ด้วย multi-platform OCI digest ใน `Dockerfile` ให้เท่ากับมาตรฐาน
เดียวกับที่ pin GitHub Actions ทุกตัว

## 5 นโยบายผลลัพธ์ของเส้นคิว

ปัญหา `_op_sanitize` เดิมคืน mapping กลับไปในผลลัพธ์ของงาน ซึ่งหมายความว่า PII จริง
เดินทางผ่านช่องผลลัพธ์ของแพลตฟอร์ม ถ้าเขาเก็บหรือ log ผลลัพธ์ คำสัญญาพังที่ฝั่งเขา
โดยที่เราตรวจสอบไม่ได้

การตัดสินใจ

- **เพิ่ม op ใหม่ชื่อ `roundtrip` และให้เป็นเส้นทางที่แนะนำ** มันปกปิด เรียก LLM
  แล้วคืนค่ากลับ จบภายในงานเดียว mapping ไม่เคยออกจาก process ทำให้คำสัญญาเป็นจริง
  เชิงโครงสร้าง (implemented ใน `app/worker/handler.py`)
- **op `sanitize` ยังคืน mapping ได้ แต่ต้องขอชัดแจ้ง** ผู้เรียกต้องใส่
  `"include_mapping": true` ใน payload ถ้าไม่ใส่ ผลลัพธ์จะไม่มีคีย์ `mapping` เลย
  (ค่าเริ่มต้นคือไม่ให้) ส่วน `restore` ยังรับ `mapping` เข้ามาเหมือนเดิม เพราะการรับ
  ค่าที่ผู้เรียกถืออยู่แล้วไม่ได้ทำให้ PII รั่วเพิ่ม
- op `analyze` และ `detect` ไม่เกี่ยวข้อง เพราะไม่เคยคืนค่าจริงอยู่แล้ว

ทางเลือกที่ปฏิเสธ คือปล่อยให้คืน mapping เหมือนเดิมแล้วเขียนกำกับว่า "ห้าม log" นั่นคือ
การฝากความปลอดภัยไว้กับความประพฤติของคนอื่น ซึ่งเราบังคับไม่ได้และพิสูจน์ไม่ได้

## 6 รหัสความผิดพลาด

| รหัส | ความหมาย |
|---|---|
| 400 | อินพุตว่าง โหมดไม่รู้จัก provider ไม่รู้จัก |
| 401 | ตั้ง `AIGUARD_API_KEY` ไว้ แต่ header ไม่มีหรือไม่ตรง |
| 404 | `session_id` ไม่รู้จักหรือหมดอายุ |
| 413 | ไฟล์เกินเพดาน (เส้นทาง PDF ซึ่งอยู่นอกหน้าสัญญา) |
| 422 | ตรวจพบ PII ระดับ FP หลงเหลือในข้อความขาออก จึงปฏิเสธ |
| 502 | LLM ต้นทางล้ม |
| 503 | ขาด dependency หรือ credential ที่จำเป็น |

ข้อความผิดพลาดและ log **ห้ามมีข้อความของผู้ใช้หรือค่า PII** ทุกกรณี (ข้อเดียวกับ VAULT-4)

## 7 การพิสูจน์

เอกสารที่ไม่มีของรันได้ประกอบ ไม่นับว่าทำแล้ว ตามกติกา audit ของโปรเจกต์นี้

- contract test ปักรูปร่าง request และ response ของทั้ง 5 endpoint
- auth test สามกรณี ไม่ตั้ง key = เปิดตามเดิม, ตั้งแล้วไม่ส่ง header = 401, ส่งถูก = 200
- ขยาย docker smoke ใน CI ให้ยก container จริงขึ้นมาแล้วยิงครบทั้ง 5 endpoint
- worker ใช้เทสเดิม บวกเทสที่ปักนโยบายข้อ 5 ว่า mapping ไม่ออกโดยไม่ได้ขอ
- เทสว่าไม่มี PII ใน log

## 8 สิ่งที่ไม่ได้สัญญา

- ไม่รับประกันความเข้ากันได้ของ endpoint ที่ไม่ได้ประกาศในข้อ 1
- ไม่รับประกันว่า mapping จะอยู่ข้ามการรีสตาร์ต container (โดยตั้งใจ)
- ไม่มี multi-tenant ไม่มี session store ร่วมข้าม instance จนกว่าจะมีผู้ใช้จริงร้องขอ
- ไม่อ้างว่าชั้น guard จับ prompt injection ได้ครบ มันเป็นชั้นเตือนแบบ
  dependency-light ที่ใช้ explicit rules ร่วมกับ bounded intent classifier
  และยังประกาศขีดจำกัดของตัวเองไว้
