# Office 2.5.0 local acceptance and promotion preparation - 2026-07-24

Status: **local XML functional acceptance and unified-package preparation
verified; packaged custom-ribbon activation remains pending**.

This record consolidates the synthetic-only Word, Excel, and PowerPoint local
acceptance evidence used to prepare the 2.5.0 unified-manifest promotion. It
does not promote local XML results into unified-manifest real-host acceptance.
No raw selection, PII value, vault mapping, provider request/response body,
credential, or screenshot is recorded here.

## Candidate and transport

- Candidate runtime and package metadata: 2.5.0.
- Backend/task pane: local FastAPI backend and HTTPS Vite task pane at trusted
  `localhost:3000`.
- Local functional transports: `office-addin/manifest.dev.xml` (Word),
  `manifest.dev.excel.xml` (Excel), and `manifest.dev.powerpoint.xml`
  (PowerPoint). Each is an acceptance-only XML manifest with a separate add-in
  ID.
- Release transport: `office-addin/manifest.json`, scoped to `document`,
  `workbook`, and `presentation` for Word, Excel, and PowerPoint.

## Headless manifest and package checks

The promotion candidate passed:

- repository validation of one unified extension; exact three-host extension
  and ribbon scopes; delegated `Document.ReadWrite.User`; trusted
  `localhost:3000`; HTTPS task-pane URL; and required icons;
- checksum-pinned official Microsoft 365 schema v1.25 validation;
- Microsoft XML schema validation for all three local acceptance manifests;
- repeatable unified ZIP construction, including root `manifest.json` and the
  declared color/outline icons byte-identical to source; and
- Office type-checking, automated tests, and production task-pane build.

These checks prove package structure. A subsequent exact-candidate acquisition
check proved Office accepted the package metadata, while custom-ribbon
activation remains a distinct client-cache/UI check.

## Local Word XML acceptance

The Word XML transport showed the AI Guard ribbon and task pane. With synthetic
content only, the following passed:

- health-ready enabled actions; backend-offline disabled actions and left the
  document unchanged;
- Detect and PDPA Analyze read without writeback;
- token and surrogate Preview/Apply/Restore returned every selected character,
  including boundary whitespace;
- changing selection after Preview cancelled Apply without changing either
  selection;
- visually uniform Thai-and-Latin text remained writable after direct-format
  checking ignored script-font fallback;
- mixed bold, size, color, or highlight remained Preview/Copy-only; and
- protected Pathumma showed masked outbound and preview-only response,
  required explicit response insertion, and reported an unused placeholder
  instead of inventing restoration.

Earlier Word unified acquisition evidence also showed ribbon/task-pane opening
after correcting `validDomains` to `localhost:3000`.

## Local Excel XML acceptance

The Excel XML transport showed the ribbon and task pane. A selected range with
synthetic text, formula, number, date, and blank cells verified that:

- Preview reported skipped non-text cells;
- Apply changed only selected text constants;
- formulas remained byte-for-byte unchanged;
- Restore worked per changed text cell in the same task-pane session; and
- changing the selected range after Preview cancelled Apply with no worksheet
  write.

## Local PowerPoint XML acceptance

The PowerPoint XML transport showed the ribbon and task pane. The following
synthetic-only cases passed:

- a uniform selected text range supported Preview, Apply, and Restore;
- an unselected placeholder stayed unchanged;
- mixed font size stayed Preview/Copy-only; and
- no text selection produced an explicit failure without a deck write.

## Unified acquisition and custom-ribbon follow-up

- `npm run start:word` registered the exact 2.5.0 package successfully and
  launched the generated Word document.
- Read-only `atk launchinfo` for manifest ID
  `9f16ea86-a09b-4ad6-95f1-c25c429594bd` reported version 2.5.0, one extension
  and ribbon scoped to `document`, `workbook`, and `presentation`, the expected
  runtime/task-pane URL, and converted XML hosts for Document, Workbook, and
  Presentation.
- Office displayed the acquired app in the Add-ins flyout, but the custom
  ribbon/task pane did not become independently observable through UI
  Automation before the client entry disappeared during cache refresh.
- The package was uninstalled cleanly with `npm run stop` after this diagnostic.

This is sufficient evidence that the release package is structurally valid and
acquirable. It is not a claim that every supported Office update channel will
surface the custom ribbon immediately, and it is not Marketplace acceptance.
The three host adapters themselves are accepted on their Microsoft-validated
local XML transports.
